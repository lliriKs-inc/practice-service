## 2026-07-18 — Экспериментальная логика заявок (замечания №5, №6 и №9)

- **Кто?** Backend и Frontend.
- **Статус решения.** Экспериментальное решение для текущего процесса практики. При необходимости его можно заменить на более строгий вариант ниже; точки отката перечислены отдельно.
- **Что исправлено?** Убрано ограничение, при котором активная заявка блокировала подачу на другие треки и когорты. Добавлены обработка PENDING-заявок при закрытии когорты, подтверждения действий администратора, причина отказа для студента и выбор рабочего трека среди одобренных заявок.
- **Файлы Backend.** backend/src/modules/application/application.service.ts, backend/src/modules/application/application.service.test.ts, backend/src/modules/cohort/cohort.service.ts, backend/src/modules/cohort/cohort.service.test.ts.
- **Файлы Frontend.** frontend/src/app/apply/[token]/page.tsx, frontend/src/app/apply/[token]/page.test.tsx, frontend/src/services/api/invitation.ts, frontend/src/lib/api/error-messages.ts, frontend/src/app/admin/applications/page.tsx, frontend/src/app/admin/applications/page.test.tsx, frontend/src/app/dashboard/applications/page.tsx, frontend/src/app/dashboard/applications/page.test.tsx, frontend/src/lib/active-application.ts, frontend/src/app/dashboard/tasks/page.tsx, frontend/src/app/dashboard/documents/page.tsx.
- **Логика подачи.** Студент может одновременно подать заявки на разные треки одной когорты и на треки других когорт. Заявки PENDING или APPROVED на других треках не блокируют новую заявку. Повторная заявка на тот же трек недоступна: это гарантирует уникальная пара пользователя и трека в базе данных.
- **Закрытие когорты.** При переводе активной когорты в CLOSED все её заявки PENDING автоматически становятся REJECTED с причиной «Когорта закрыта до рассмотрения заявки». Одобренные заявки не изменяются.
- **Решение администратора.** Администратор может одобрить несколько заявок одного студента на разные треки одной когорты. Другие заявки автоматически не отклоняются: окончательный выбор остаётся у студента. Перед одобрением и отклонением выводятся модальные окна в стиле сайта; при отклонении администратор может указать причину.
- **Действия студента.** Причина отклонения отображается в «Моих заявках». Среди одобренных заявок студент нажимает «Выбрать этот трек» и подтверждает действие в модальном окне. До даты начала практики выбор можно менять неограниченно; после начала кнопка блокируется и отображается «Выбор закреплён». После подтверждения студент остаётся на странице заявок.
- **Влияние выбора трека.** Выбор хранится локально в браузере под ключом active_application_id. Он определяет трек в разделах «Задачи» и «Документы» и показывается в их заголовках. Выбор не меняет серверные статусы и права: все одобренные заявки остаются одобренными.
- **Что было раньше?** Одна активная заявка блокировала новые, закрытие могло оставлять PENDING-заявки, студент не видел причину отказа, а подтверждение административных действий было неоднородным. Промежуточную логику, где одобрение одной заявки отклоняло остальные, отменили по результатам обсуждения.
- **Как протестировано?** Ручная проверка подтвердила: несколько заявок, автоматическое отклонение PENDING при закрытии, несколько одобренных заявок, оба модальных окна, причину отказа, выбор и смену трека до старта, блокировку после старта, отображение выбранного трека в задачах и документах, отсутствие перехода в задачи после выбора. Успешно выполнены backend typecheck и 20 профильных backend-тестов, frontend typecheck, ESLint и 20 профильных frontend-тестов.
- **Последствия и ограничения.** Выбор привязан к устройству и браузеру: после очистки localStorage или на другом устройстве будет выбран первый доступный одобренный трек. Для единого выбора на всех устройствах потребуется серверное поле и API. Вместимость треков текущей моделью не ограничивается.
- **Точки отката.** Для возврата к единственному одобрению на когорту нужно восстановить проверку и транзакционное отклонение остальных заявок в ApplicationService.updateStatus, а во Frontend убрать выбор трека. Для возврата блокировки подачи нужно вернуть проверку активных заявок в странице подачи и invitation API. Автоматическое отклонение PENDING при закрытии можно оставить или убрать независимо.

### Альтернативный строгий вариант — не реализован

- У студента только одна заявка в одну когорту, на выбранный трек; после REJECTED он может подать на другой трек; у треков есть ограниченные места.
- Он не выбран сейчас, потому что не соответствует текущей практике: сильный студент может быть одобрен на нескольких направлениях и сам выбрать итоговое.
- Для перехода потребуются миграция и уникальность по пользователю и когорте вместо пользователя и трека, логика повторной подачи после REJECTED, вместимость трека и транзакционная проверка мест при одобрении, изменения интерфейсов и пересмотр localStorage-выбора.
- Такой переход меняет бизнес-процесс и должен быть отдельной согласованной задачей с миграцией данных и повторной проверкой подачи, закрытия и одобрения.

## 2026-07-20 — Замечание №12: скачивание отчёта студента в Admin

- **Кто?** Backend и Frontend.
- **Что исправлено?** В Admin → Документы добавлена и доведена до рабочего состояния кнопка «Скачать отчёт». Отчёт скачивается защищённым запросом с проверкой роли Admin и принадлежности заявки выбранной когорте.
- **В каких файлах?** frontend/src/app/admin/documents/page.tsx, frontend/src/lib/api/download.ts, backend/src/modules/documents/documents.controller.ts, backend/src/modules/documents/document-file.controller.ts, backend/src/modules/documents/document-file.service.ts, backend/src/modules/documents/document-file.routes.ts, backend/src/shared/storage/local-storage.service.ts, backend/src/shared/storage/file-download.handler.ts, backend/src/app.ts и соответствующие backend-тесты.
- **Что именно изменено?** Кнопка скачивания показывается только у заявки, для которой существует отчёт. Для новых отчётов имя проходит ту же нормализацию multipart UTF-8/Latin-1, что и решение тестового задания. Из-за ограничения «без изменения БД» исходное имя не добавлялось в модель Report: storage key отчёта содержит ASCII-safe base64url-представление имени (`<uuid>__name_<encoded>.<ext>`), а backend восстанавливает имя при выдаче файла. Кириллица не помещается напрямую в URL или в небезопасный HTTP-заголовок.
- **Что исправлено в download API?** В отдельном маршруте отчёта `DocumentFileController` убрана прямая передача Unicode в `Content-Disposition`, которая вызывала Node.js `TypeError: Invalid character in header content`. Теперь `Content-Disposition` содержит только ASCII-safe fallback, а исходное имя передаётся в `X-Download-Filename` в URL-кодировке. CORS открывает этот заголовок Frontend. Общий Frontend-механизм защищённого скачивания сначала декодирует `X-Download-Filename`, затем использует стандартный fallback.
- **На что влияет?** Admin может скачать PDF/DOC/DOCX без публичной ссылки и без потери имени. Доступ по-прежнему проверяется на backend; Student может скачивать свой отчёт тем же защищённым способом. Решения без отчёта кнопку не показывают. База данных и Prisma-схема не изменялись.
- **Что получилось?** Раньше кнопка отсутствовала или файл скачивался без расширения, с UUID-именем, mojibake либо завершался HTTP 500 из-за кириллического `Content-Disposition`. Теперь новый отчёт с именем `Анализ материалов Nike Dunk SB.docx` скачивается с тем же читаемым именем и расширением. Старые UUID-файлы, загруженные до сохранения имени в storage key, не могут восстановить исходное имя задним числом.
- **Как протестировано?** Пользователь подтвердил ручную проверку скачивания отчёта Admin. Дополнительно проверены имена `Лабораторная работа 4 МсИС.pdf` и `Анализ материалов Nike Dunk SB.docx`, устранён HTTP 500 `Invalid character in header content`.
